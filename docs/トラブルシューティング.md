## 1. node_modules が大きすぎて GitHub にプッシュできない

### どんなエラーが出たか

```
remote: error: File apps/portal/node_modules/... is 100.00 MB; this exceeds GitHub's file size limit of 100.00 MB
```

### なぜ起きたのか

`node_modules` というフォルダには、アプリが動くために必要な「部品」がたくさん入っています。  
たとえば Next.js や React など、他の人が作ったプログラムの集まりです。

GitHub（コードを保存する場所）には **1ファイル100MB以上は送れない** というルールがあります。  
`node_modules` の中には巨大なファイルが含まれていることがあり、このルールに引っかかります。

さらに、`.gitignore`（「これは送らなくていい」という設定ファイル）を作る前に `git add .` をしてしまうと、`node_modules` まるごとがコミット対象に入ってしまいます。

### どうやって解決したか

1. `.gitignore` ファイルを作って、`node_modules/` や `.next/` などを除外対象にする
2. 一度 `.git` フォルダ（git の記録帳）を削除して、最初からやり直す
3. `git init` で再スタートし、`.gitignore` が効いた状態でコミット

```bash
rm -rf .git
git init
git add .gitignore
git add .
git commit -m "Initial commit"
git push
```

### 教訓

> **コードを GitHub に送る前に、必ず `.gitignore` を確認する。**  
> `node_modules/` は毎回 `npm install` で自動的に作られるので、GitHub には送る必要がありません。

## 2. Vercel のリージョンエラー（nrt1 が使えない）

### どんなエラーが出たか

```
Invalid region selector: "nrt1". Valid regions: iad1, sfo1, hnd1, ...
```

### なぜ起きたのか

Vercel（クラウドにアプリを公開するサービス）には、サーバーの「場所（リージョン）」を指定できます。  
日本に近い場所を指定すると速くなります。

`nrt1` は昔の日本リージョンのコードですが、今は **`hnd1`（Haneda＝羽田）** に変わっています。  
古いコードをそのまま使ったため、「そんなリージョンはない」とエラーが出ました。

### どうやって解決したか

`vercel.json` の `regions` を `"nrt1"` から `"hnd1"` に変更する。

```json
{
  "regions": ["hnd1"]
}
```

## 3. ローカルの .env を Vercel 用の URL に書き換えてしまった

### どんな問題が起きたか

ローカルでアプリを動かしたとき、ログインができなくなった。  
`NEXTAUTH_URL` が Vercel の URL（`https://...vercel.app`）になっていたため、  
手元のパソコンでアクセスしたときに「URLが違う」とエラーになった。

### なぜ起きたのか

`NEXTAUTH_URL` とは「このアプリはどこで動いているか」を Next-Auth に教える設定です。  

- **ローカル開発時** → `http://localhost:3010`
- **Vercel本番** → `https://xxx.vercel.app`

Vercel 用の設定値をうっかり `.env`（ローカルの設定ファイル）に書いてしまうと、  
ローカルで動かしたときに「あれ、localhost:3010 のはずがVercel URLになってる」と混乱します。

### どうやって解決したか

`.env` を元のローカル用の値に戻す。

```env
NEXTAUTH_URL="http://localhost:3010"
DATABASE_URL="file:./dev.db"
```

Vercel 用の環境変数は、Vercel のダッシュボードの「Environment Variables」から設定する（`.env` には書かない）。

### 教訓

> **`.env` はローカル専用。Vercel の設定はVercelのダッシュボードで管理する。**

## 4. Vercel でログインできない（メールアドレスかパスワードが違います）

### どんな問題が起きたか

Vercel にデプロイしたポータルに `admin@demo.local / admin1234` でログインしようとしたが、  
「メールアドレスかパスワードが違います」というエラーが出た。

### なぜ起きたのか

ローカルでは SQLite（手元のパソコンの中にあるデータベース）を使っています。  
Vercel 本番では PostgreSQL（Neon という外部サービス）を使います。

デプロイしたばかりのとき、Neon のデータベースは **空っぽ** です。  
ユーザーのデータ（メールアドレス・パスワード）が入っていないので、ログインできませんでした。

### どうやって解決したか

シード（初期データ投入）スクリプトを手動で実行してユーザーを作成する。

```bash
DATABASE_URL="postgresql://..." npx tsx prisma/seed.ts
```

このコマンドで、Neon の PostgreSQL に以下のユーザーが作成される：
- `admin@demo.local` / `admin1234`
- `office@demo.local` / `office1234`
- `field@demo.local` / `field1234`

### 教訓

> **クラウドにデプロイした直後は、データベースが空。初期データを必ず投入する。**

## 5. アプリの中のリンクをクリックするとトップページに戻らない / 404 になる

### どんな問題が起きたか

`/apps/work` の中で `<a href="/">` をクリックすると、`/` に遷移してしまい、  
`/apps/work/` のトップに戻れなかった。

### なぜ起きたのか

各アプリには `basePath` という設定がついています。  
たとえば excel-reduction には `basePath: '/apps/work'` という設定があります。

`basePath` が設定されているとき、Next.js の `<Link href="/">` は自動的に `/apps/work/` に変換されます。  
しかし、通常の HTML タグ `<a href="/">` は変換されず、そのまま `/` になってしまいます。

### どうやって解決したか

すべての `<a href="...">` を Next.js の `<Link href="...">` に変更する。

```tsx
// 修正前（動かない）
<a href="/">トップへ</a>

// 修正後（動く）
import Link from 'next/link'
<Link href="/">トップへ</Link>
```

## 6. 4つのアプリが開けない（portal から遷移できない）

### どんな問題が起きたか

ポータルのホーム画面からアプリカードをクリックしても、アプリが開かなかった。  
ngrok を使った公開 URL でアクセスしていたため、カードのリンクが `http://localhost:3000` のような  
ローカルアドレスになっており、外部からアクセスできなかった。

### なぜ起きたのか

もともとアプリカードのリンクは「作業入力アプリは localhost:3000 で動いている」という前提で  
ハードコードされていました。ngrok などで外部公開するとき、そのアドレスは通じません。

### どうやって解決したか

**Next.js のリバースプロキシ機能（rewrites）** を使いました。

「ポータルの `/apps/work/...` へのリクエストは、こっちに転送する」という設定です。

```js
// portal/next.config.mjs
async rewrites() {
  return [
    { source: '/apps/work/:path*', destination: 'http://localhost:3000/apps/work/:path*' },
    // ...
  ]
}
```

こうすることで、ポータルの URL（例：`https://xxx.ngrok.app/apps/work/`）にアクセスすれば、  
内部でローカルの 3000 番ポートに転送してくれます。外部からは URL が1つに見えます。

## 7. TypeScript エラー: 'frequency' is specified more than once

### どんなエラーが出たか

```
TypeScript error: 'frequency' is specified more than once, so this usage will be overwritten.
```

### なぜ起きたのか

オブジェクトを作るとき、同じプロパティを2回書くとエラーになります。

```ts
// エラー: frequency が2回ある
const obj = { frequency: 'daily', ...t, frequency: t.frequency ?? 'daily' }
```

### どうやって解決したか

スプレッド演算子（`...t`）を先に書いて、後から上書きする形にする。

```ts
// OK: スプレッドで t の全プロパティを展開した後に frequency だけ上書き
const obj = { ...t, frequency: t.frequency ?? 'daily' }
```

こうすると「t の中に frequency があればそれを使い、なければ 'daily' を使う」という意味になります。

## 8. next.config.ts が Vercel で認識されない

### どんな問題が起きたか

`next.config.ts` という名前のファイルを作ったが、Vercel でビルドしたときにエラーになった。

### なぜ起きたのか

Next.js 14 では設定ファイルとして `next.config.js` または `next.config.mjs` しか対応していません。  
`.ts`（TypeScript）の設定ファイルはサポート外でした。

### どうやって解決したか

`next.config.ts` を `next.config.mjs` に名前変更する。  
（`.mjs` は「モジュール形式の JavaScript ファイル」という意味です）

## 9. redirect のパスに route group 名が入ってしまう

### どんな問題が起きたか

Next.js の App Router で `(portal)` という route group を使っていたが、  
`redirect('/(portal)/home')` と書いてしまい、リダイレクトが正しく動かなかった。

### なぜ起きたのか

Next.js の「route group」は、フォルダを `()` で囲むと **URL に影響しない** 特別な機能です。  
つまり `(portal)/home/page.tsx` のファイルは、URL では `/home` になります。  
`/(portal)/home` という URL は実際には存在しません。

### どうやって解決したか

```ts
// 間違い
redirect('/(portal)/home')

// 正解
redirect('/home')
```

## 10. ダッシュボードのデータが Vercel で取得できない（ファイルが見つからない）

### どんな問題が起きたか

Vercel にデプロイしたダッシュボードで、作業実績や点検記録のデータが表示されなかった。

### なぜ起きたのか

ローカルでは、ダッシュボードは兄弟フォルダ（`../excel-reduction/records.json` など）から  
直接ファイルを読み込んでいました。

しかし **Vercel のサーバーはファイルシステムが読み取り専用** で、しかも各アプリは別々のサーバーに  
デプロイされます。「隣のフォルダ」という概念がありません。

### どうやって解決したか

`process.env.VERCEL` が設定されているとき（＝Vercel上で動いているとき）は、  
ファイルを読む代わりに **HTTP APIを呼び出す** ように変更しました。

```ts
async function fetchAllWorkRecords(): Promise<WorkRecord[]> {
  if (process.env.VERCEL) {
    // Vercel上: APIを呼び出してデータ取得
    const base = process.env.APP_WORK_URL ?? ''
    const res = await fetch(`${base}/apps/work/api/records`)
    return res.json()
  }
  // ローカル: ファイルから直接読み込み
  const p = path.join(dir, 'records.json')
  return JSON.parse(fs.readFileSync(p, 'utf-8'))
}
```

## 11. async 関数に await をつけ忘れた（データが空になる）

### どんな問題が起きたか

ダッシュボードの API ルートでデータが空（`[]`）になっていた。

### なぜ起きたのか

データ取得関数を `async`（非同期）に変更したあと、呼び出し側で `await` を書き忘れていました。

```ts
// 間違い: await がない → Promise オブジェクトが返るだけでデータが入っていない
const workRecords = queryWorkRecords({ dateFrom, dateTo })
```

JavaScript では `async` 関数は「後で結果をくれる約束（Promise）」を返します。  
`await` を書くことで「結果が来るまで待つ」という意味になります。

### どうやって解決したか

```ts
// 正解: await をつける
const workRecords = await queryWorkRecords({ dateFrom, dateTo })

// 複数ある場合は Promise.all で並列取得（速い）
const [workRecords, inspRecords] = await Promise.all([
  queryWorkRecords({ dateFrom, dateTo }),
  queryInspectionRecords({ dateFrom, dateTo }),
])
```

### 教訓

> **`async` 関数を呼ぶときは必ず `await`。これを忘れるとデータが空のままになる。**

## 12. サブアプリを直接 URL で開くと 404 になる（basePath の仕組み）

### どんな問題が起きたか

`https://gyomu-work.vercel.app` にアクセスしたら 404 になった。

### なぜ起きたのか

各サブアプリには `basePath` という設定がついています。  
たとえば作業入力アプリは `basePath: '/apps/work'` と設定されています。

これは「このアプリは `/apps/work` という場所から始まる」という宣言です。  
だからトップページ（`/`）には何も置かれておらず、404 になります。

```
❌ https://gyomu-work.vercel.app/          ← 何もない → 404
✅ https://gyomu-work.vercel.app/apps/work ← ここが実際のトップページ
```

### なぜ basePath をつけているのか

ポータルが「`/apps/work/...` へのアクセスを gyomu-work に転送する」という仕組みを使っているからです。  
転送先も同じパス（`/apps/work`）で動いている必要があります。

### 各アプリの正しい直接アクセス URL

| アプリ | 直接アクセスURL |
|--------|----------------|
| 作業入力 | `https://gyomu-work.vercel.app/apps/work` |
| 点検 | `https://gyomu-inspection.vercel.app/apps/inspection` |
| ダッシュボード | `https://gyomu-dashboard.vercel.app/apps/dashboard` |
| 事例管理 | `https://gyomu-case.vercel.app/apps/case` |

> 通常はポータル経由でアクセスするため、直接 URL を使う必要はありません。

## 13. 環境変数名が複数あって混乱する（NEXT_PUBLIC_ / INTERNAL_ / APP_）

### どんな問題が起きたか

環境変数の名前が似ていて、どれをどこに設定すれば良いか混乱した。

### 整理

このシステムには3種類の環境変数があります：

| 変数名のパターン | 設定するプロジェクト | 用途 |
|----------------|-------------------|------|
| `INTERNAL_APP_*_URL` | **portal** | ポータルがサブアプリに転送するための URL |
| `APP_WORK_URL` / `APP_INSPECTION_URL` | **dashboard** | ダッシュボードがデータを取得するための URL |
| `NEXT_PUBLIC_APP_*_URL` | （使わない） | 初期設計の名残。現在は不要 |

### ポイント

**`INTERNAL_APP_*_URL`** の値には **ドメインだけ** 入れる。パスは不要。

```
✅ INTERNAL_APP_WORK_URL = https://gyomu-work.vercel.app
❌ INTERNAL_APP_WORK_URL = https://gyomu-work.vercel.app/apps/work  ← /apps/work は不要
```

ポータルの設定が自動で `/apps/work/:path*` を付け足すので、ドメインだけで OK です。
