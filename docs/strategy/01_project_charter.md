# 中小企業向けマイクロアプリ事業 企画書 / ソース資料

## 1. 目的

本プロジェクトの目的は、  
**父親の会社を最初の実証環境として使い、製造業系の中小企業向けに小型業務アプリを開発し、周辺企業へ横展開することで、月5万〜20万円の副収入を作ること** である。

この挑戦で重視する順序は以下。

1. 父親の会社で実証して横展開する
2. 月5万〜20万円の副収入化
3. 自分のスキル資産を増やす
4. 最速で初売上を出す
5. 将来大きく伸びる事業の種を作る
6. 中小企業向けの実績・信用を作る

---

## 2. このプロジェクトの前提条件

### 強み
- 要件定義・構造化・実装判断が強い
- Claude Code を使った高速実装が可能
- ChatGPT を使った戦略整理・仮説整理が可能
- 父親の会社という一次情報源 / PoC環境がある
- 改善サイクルを高頻度で回せる
- 製造業・現場・業務効率化文脈との相性が良い

### 弱み
- マーケティング知識がほぼない
- 顔出し営業は当面しない
- 平日の稼働は1時間程度と限られる
- 高リスク領域はまだ避ける必要がある

### 制約
- 予算は月1〜2万円
- 実装はコードメイン
- 法務・個人情報・会計・労務の深い領域は初期に避ける
- 実証先はまず父親の会社
- ターゲット市場は製造業の中小企業とその周辺企業

---

## 3. 結論：この挑戦の勝ち筋

結論から言うと、この挑戦の勝ち筋は  
**「量産」ではなく「父親の会社で刺さった小さな業務改善を、同業他社にも通じる定型商品に変換すること」**
にある。

つまり、成功条件は次の通り。

### 勝ち筋の本質
1. 現場の小さく強い痛みを見つける  
2. 1週間以内にMVPを作る  
3. 父親の会社で実運用して数字を取る  
4. その結果を1枚の事例にまとめる  
5. 周辺企業へ「同じ悩みありませんか」で横展開する  

### 重要な再定義
「1日1個作る」は、そのままだと失敗率が高い。  
正しくは、以下の意味に変換する。

- 1日1個 = **1日1仮説 / 1日1検証単位**
- 完成品を毎日量産するのではない
- 毎日「痛みの確認」「画面ラフ」「業務フロー整理」「簡易MVP」「提案文」「導入テスト」のどれかを進める
- 売れたものだけを育てる

この再定義をしないと、作ること自体が目的化し、収益に繋がらない。

---

## 4. やるべきでないこと

### 初期に避けるべきテーマ
- 給与計算
- 勤怠の厳密管理
- 会計・請求の法制度依存領域
- 個人情報を重く扱うシステム
- 大規模な基幹連携前提のSaaS
- 説明に30分以上かかるプロダクト
- 現場で毎日入力を強制するアプリ
- AIであること自体を売りにするアプリ

### なぜ避けるべきか
- リスクが高い
- 導入に慎重になりやすい
- 検証サイクルが遅くなる
- 現場で使われないと価値が出ない
- 副業1時間運用と相性が悪い

---

## 5. 最初に狙うべき領域

最初は、**「現場の面倒な確認・集計・伝達・見える化」** に絞る。

### 理由
- 法務リスクが低い
- 導入説明が短い
- 効果が見えやすい
- 父親の会社で実証しやすい
- 同業他社にも転用しやすい

### 優先すべきテーマ
1. 日報・点検・報告の入力簡略化
2. Excel転記や手作業集計の削減
3. 紙・LINE・口頭運用の整理
4. 見積・発注前段階の情報整理
5. 在庫 / 備品 / 作業状況の軽い見える化

---

## 6. 推奨する事業モデル

最初の推奨は、**買い切り小型ツール + 軽保守** である。  
その後、使われ方によって月額へ移行する。

### 理由
- 中小企業は月額に警戒しやすい
- 「まず一回入れてみる」が通りやすい
- 副業での運用負荷が低い
- 初回売上が立ちやすい
- 実績ができると月額化しやすい

### 推奨価格帯
- 簡易ツール：3万円〜5万円
- 現場向け小型アプリ：5万円〜10万円
- 管理画面付きの実用版：10万円〜20万円
- 保守：月5千円〜2万円
- 追加改修：都度見積

### 月額SaaSに向く条件
- 毎月繰り返し使う
- 複数人で継続利用する
- データ蓄積に価値がある
- 使うほど手放しにくくなる

---

## 7. 予算1〜2万円の使い道

AIサブスク除外なら、優先順位は以下。

1. 独自ドメイン
2. LP公開用の低コストホスティング
3. フォーム / 予約 / 問い合わせ導線
4. 事例PDF / 提案資料の整備
5. 必要最小限の分析ツール

最初は広告に使わない。  
初期の売上は **父親の会社 → 紹介 → 周辺企業** で作る方が合理的。

---

## 8. ノーコードの扱い

ここでいうノーコードとは、例えば以下。
- Notion
- Airtable
- Glide
- Softr
- Google Apps Script ベースの軽実装
- Zapier / Make などの連携

ただし、あなたの強みはコード実装なので、基本方針は次の通り。

### 方針
- **コアはコードで作る**
- ノーコードは仮説検証や管理画面代用で使う
- 本番の価値部分までノーコード依存しすぎない

---

## 9. 成功指標

### 30日以内
- 父親の会社でヒアリング 5件以上
- アプリ候補 10個以上
- MVP 2〜3本
- 実運用PoC 1本
- 改善前後の数字 1件
- 横展開用の事例シート 1枚

### 90日以内
- 初売上 1件以上
- 有料導入 2〜3社
- 月5万円到達の見込み
- 再利用できるテンプレ基盤 2本以上

---

## 10. このプロジェクトの基本原則

1. 作る前に、誰の何分を削るかを言語化する  
2. 父親の会社で使えないものは、他社でも売れない可能性が高い  
3. AI感より、現場の手間削減を売る  
4. 画面の美しさより、導入後1週間で効果が出ることを優先  
5. 事例化できないPoCは半分失敗  
6. 「便利そう」ではなく「すでに困っている」を狙う
